# Architecture

- ExoPattern is a Python monorepo with two runtime entry points: Flask in [`backend/app.py`](/home/arnavbansal/ExopatternV3-copy/backend/app.py) and Streamlit in [`frontend/app.py`](/home/arnavbansal/ExopatternV3-copy/frontend/app.py).
- The core pipeline is lightcurve ingestion -> preprocessing -> sliding-window feature extraction -> anomaly scoring -> point/event aggregation -> visualization or API response.
- The codebase centers on a shared ML layer in [`backend/ml/`](/home/arnavbansal/ExopatternV3-copy/backend/ml) that is reused by the API, Streamlit UI, and experiments.
- Data loading is split by format and source: local FITS/CSV parsing in [`backend/data/loader.py`](/home/arnavbansal/ExopatternV3-copy/backend/data/loader.py), remote MAST / ephemeris acquisition in [`backend/data/acquisition.py`](/home/arnavbansal/ExopatternV3-copy/backend/data/acquisition.py).
- The model abstraction is a thin wrapper layer around scikit-learn and TensorFlow objects, exposed through [`backend/ml/model_registry.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/model_registry.py).
- Classical anomaly detection is the primary production path; the ensemble logic is implemented in [`backend/ml/models.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/models.py).
- Feature engineering is window-based and lives in [`backend/ml/preprocessing.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/preprocessing.py), with feature names centralized in [`backend/ml/feature_names.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/feature_names.py).
- Training persistence is file-system based, with artifacts stored under [`backend/models/`](/home/arnavbansal/ExopatternV3-copy/backend/models) and larger pre-trained bundles under [`artifacts/models/`](/home/arnavbansal/ExopatternV3-copy/artifacts/models).
- The Flask API is stateful within process: it loads a detector at startup, then mutates the in-memory `detector` object during training endpoints in [`backend/api/routes.py`](/home/arnavbansal/ExopatternV3-copy/backend/api/routes.py).
- The Streamlit app is also stateful through `st.session_state`, mainly storing detector, preprocessor, and the latest analysis results in [`frontend/app.py`](/home/arnavbansal/ExopatternV3-copy/frontend/app.py).
- The project supports two analysis modes: interactive single-file analysis in the UI and batch experiment execution in [`experiments/run_experiments.py`](/home/arnavbansal/ExopatternV3-copy/experiments/run_experiments.py).
- Experiment configuration is centralized in [`experiments/config.py`](/home/arnavbansal/ExopatternV3-copy/experiments/config.py), while grid search lives in [`experiments/hyperparameter_tuning.py`](/home/arnavbansal/ExopatternV3-copy/experiments/hyperparameter_tuning.py).
- The current architecture uses a “window model, point projection” pattern: window predictions are mapped back to cadence-level masks using [`backend/ml/models.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/models.py).
- API endpoints are narrow and task-based: `/api/analyze`, `/api/train`, `/api/train/synthetic`, `/api/export`, `/api/models`, and `/api/evaluate` in [`backend/api/routes.py`](/home/arnavbansal/ExopatternV3-copy/backend/api/routes.py).
- The UI is a multi-tab dashboard rather than a page-router app; its tabs are analysis, model comparison, periodogram, model inspector, and reference content in [`frontend/app.py`](/home/arnavbansal/ExopatternV3-copy/frontend/app.py).
- Model selection is registry-driven, not hard-coded, so adding a new detector generally means adding a factory in [`backend/ml/model_registry.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/model_registry.py) and exposing it in the UI.
- Classical and deep models share a common concept of `fit`, `predict`, and `score_samples`, but deep models are loaded conditionally because TensorFlow is optional in [`backend/ml/deep_models.py`](/home/arnavbansal/ExopatternV3-copy/backend/ml/deep_models.py).
- Publication outputs are treated as first-class artifacts: figures, tables, and JSON results are generated under [`results/`](/home/arnavbansal/ExopatternV3-copy/results) by the experiment code.
- Kaggle notebooks are part of the architecture, not an afterthought; they are the heavy-compute training and benchmarking path referenced by the README and run from [`notebooks/kaggle/`](/home/arnavbansal/ExopatternV3-copy/notebooks/kaggle).
- The poster in [`poster/index.html`](/home/arnavbansal/ExopatternV3-copy/poster/index.html) is a separate presentation layer, not wired into runtime code.
- Dependency boundaries are implicit rather than enforced by a package manager with lockfile, so shared utilities are imported directly from `backend.*` modules across app, API, and experiments.
- The architecture currently favors “shared library + thin frontends + notebook workflows,” which is good for research code but requires discipline to avoid drift between production, notebook, and artifact states.
